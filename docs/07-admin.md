# 7. لوحة الإدارة (Admin)

المصدر: `apps/api/src/users/users.service.ts`،
`apps/api/src/users/users.controller.ts`، `apps/api/prisma/seed.ts`،
وفي الواجهة `apps/web/src/features/admin/UsersAdminPage.tsx`.

## النطاق

مسارات الإدارة كلها تحت `admin/users` ومحميّة بـ `@UseGuards(JwtAuthGuard, AdminGuard)`
— أي تتطلّب مستخدمًا دوره **ADMIN** على مستوى النظام (انظر
[`04-authorization.md`](./04-authorization.md)).

| المسار | الوظيفة |
|---|---|
| `GET /admin/users` | سرد المستخدمين مع بحث وتقسيم صفحات |
| `PATCH /admin/users/:id/role` | تغيير دور المستخدم (USER/ADMIN) |
| `PATCH /admin/users/:id/status` | تفعيل/تعطيل المستخدم |

## سرد المستخدمين (List)

- يدعم بحثًا نصيًا (غير حسّاس لحالة الأحرف) على **البريد** و**الاسم المعروض**.
- يدعم تقسيم الصفحات: `page` (افتراضيًا 1) و `pageSize` (افتراضيًا 20، وحتى 100).
- يُرجِع لكل مستخدم **عدد اللوحات** التي هو عضو فيها (`boardCount`)، والإجمالي للترقيم.

## تغيير الدور (Update Role) — القواعد الحاكمة

كل العمليات تجري داخل **معاملة**، وتُفحص المتغيّرات مقابل **قاعدة البيانات** لا رمز
الدخول (قد يحمل دورًا قديمًا). عند **التنزيل من ADMIN إلى USER**:

1. **لا يمكن للمدير تنزيل نفسه** → `Forbidden`.
2. **لا يمكن تنزيل آخر مدير مُفعّل** في النظام → `Conflict` (قاعدة "آخر مدير").
3. تُبطَل رموز تحديث المستخدم المُنزَّل فورًا، بحيث ينتهي وصوله المرتفع خلال مدة رمز
   الدخول لا مدة رمز التحديث.

إن كان الدور المطلوب مطابقًا للحالي تُرجَع البيانات دون تغيير.

## تفعيل/تعطيل المستخدم (Update Status) — القواعد الحاكمة

عند **التعطيل** (`isActive = false`):

1. **لا يمكن للمدير تعطيل نفسه** → `Forbidden`.
2. **لا يمكن تعطيل آخر مدير مُفعّل** → `Conflict`.
3. تُبطَل رموز تحديث المستخدم المُعطَّل. (تسجيل الدخول أصلًا مرفوض للمعطَّلين، وإبطال
   رموز التحديث يحدّ من عمر رموز الدخول الحيّة حتى انتهائها.)

## قاعدة "آخر مدير مُفعّل" (Last Active Admin)

`assertNotLastActiveAdmin` تمنع أي عملية (تنزيل أو تعطيل) قد تترك النظام **بلا أي مدير
مُفعّل**، وذلك بعدّ المديرين المُفعّلين الآخرين؛ إن كان العدد صفرًا تُرفض العملية. هذا يضمن
عدم فقدان القدرة على إدارة النظام نهائيًا.

## بذرة المدير الافتراضي (Seed)

`prisma/seed.ts` تُنشئ (أو تُصلِح) مستخدمًا مديرًا افتراضيًا:
- تُقرأ بياناته من متغيّرات البيئة `SEED_ADMIN_EMAIL` / `SEED_ADMIN_PASSWORD` /
  `SEED_ADMIN_NAME`، وبقيم افتراضية للتطوير المحلي.
- تُستخدم عملية `upsert` تُعيد **تأكيد** الدور `ADMIN` والتفعيل عند كل تشغيل، بحيث يبقى
  دائمًا حساب مدير صالح واحد على الأقل حتى لو جرى تنزيله أو تعطيله سابقًا.

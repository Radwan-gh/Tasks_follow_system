# 5. منطق اللوحات والقوائم والبطاقات

المصدر: `apps/api/src/boards/boards.service.ts`،
`apps/api/src/lists/lists.service.ts`، `apps/api/src/cards/cards.service.ts`،
وأشكال الطلبات في `packages/types/src/requests.ts`.

## واجهة الـ API (نظرة سريعة)

كل المسارات تتطلّب مصادقة (Bearer JWT) ما عدا `/auth/*`.

- **اللوحات:** `GET/POST /boards`، `GET/PATCH/DELETE /boards/:id`،
  `POST/DELETE /boards/:id/members[/:userId]`
- **القوائم:** `POST /boards/:boardId/lists`، `PATCH/DELETE /lists/:id`
- **البطاقات:** `POST /lists/:listId/cards`، `GET/PATCH/DELETE /cards/:id`

## اللوحات (Boards)

### الإنشاء
- ينشئ اللوحة ويُسجّل منشئها كعضو **OWNER** في نفس العملية.

### القائمة (List for user)
- تُرجَع فقط اللوحات التي المستخدم **عضو** فيها و**غير مؤرشفة**، مرتّبة بالأحدث تحديثًا.

### التفاصيل (Detail)
- يتطلّب عضوية (MEMBER على الأقل).
- يُرجِع اللوحة مع أعضائها وقوائمها **غير المؤرشفة** مرتّبة بالموضع، وداخل كل قائمة
  بطاقاتها **غير المؤرشفة** مرتّبة بالموضع.

### التعديل والحذف
- تعديل الاسم/الوصف: MEMBER. أرشفة اللوحة: OWNER. الحذف: OWNER (انظر
  [`04-authorization.md`](./04-authorization.md)).

### إدارة الأعضاء
- **الإضافة (OWNER):** يُبحث عن المستخدم بالبريد؛ يُرفض إن لم يوجد، أو إن كان عضوًا
  بالفعل. يُضاف بدور **MEMBER**.
- **الإزالة (OWNER):** يُرفض إن لم توجد العضوية، و**يُمنع إزالة عضو دوره OWNER**.

## القوائم (Lists)

### الإنشاء
- يتطلّب عضوية. يُحسب موضع القائمة الجديدة بحيث تُضاف **في نهاية** اللوحة، عبر قراءة
  آخر قائمة غير مؤرشفة وتوليد مفتاح بعدها (`nextListPosition`).

### التعديل وإعادة الترتيب
- يمكن تعديل الاسم أو حالة الأرشفة.
- إعادة الترتيب تتم عبر حقل `move: { beforeId, afterId }` (انظر أدناه "شكل طلب النقل").
- يُتحقّق أن معرّفات الجيران المُرسلة تنتمي فعلًا **لنفس اللوحة** وأنها ليست القائمة
  نفسها المُحرَّكة، وإلا يُرفض بـ "Invalid move target".

### الحذف
- يتطلّب عضوية؛ يحذف القائمة وبطاقاتها (حذف متسلسل).

## البطاقات (Cards)

### الإنشاء
- تحت قائمة محدّدة؛ يتطلّب عضوية في لوحة القائمة.
- تُضاف **في نهاية** القائمة (`nextCardPosition`)، ويُسجَّل منشئها في `createdById`،
  ويُنسَخ `boardId` من القائمة.

### التعديل والنقل — أهم منطق في هذا الملف
دالة `CardsService.update` تغطّي عدّة حالات في آنٍ واحد:

1. **تعديل الحقول فقط** (عنوان/وصف/تاريخ استحقاق/أرشفة) دون نقل.
2. **النقل داخل نفس القائمة** عبر `move: { beforeId, afterId }`.
3. **النقل إلى قائمة أخرى** عبر `targetListId`:
   - يُتحقّق أن القائمة الهدف **تنتمي لنفس اللوحة**؛ يُرفض النقل بين لوحتين مختلفتين
     ("Cannot move a card to a list on a different board").
   - إن أُرسلت معرّفات الجيران يُتحقّق أنها تنتمي **للقائمة الهدف** وأنها ليست البطاقة
     نفسها.
   - إن نُقلت لقائمة أخرى **دون تحديد جيران**، تُوضع **في نهاية** القائمة الهدف.

كل حساب للموضع يجري **داخل معاملة Prisma واحدة**، بإعادة قراءة مواضع الجيران طازجةً ثم
حساب المفتاح الجديد على الخادم. النتيجة: **موضع قديم أرسله العميل لا يمكن أبدًا أن يُفسد
الترتيب** (تفاصيل الخوارزمية في [`06-ordering.md`](./06-ordering.md)).

### الحذف
- يتطلّب عضوية في لوحة البطاقة.

## شكل طلب النقل (Move Request) — قاعدة موحّدة

كل طلبات إعادة الترتيب (للقوائم وللبطاقات) تشير إلى **معرّفات الجيران** لا إلى سلاسل
الموضع الخام:

```
PATCH /lists/:id   { move: { beforeId, afterId } }
PATCH /cards/:id   { targetListId?, move: { beforeId, afterId } }
```

- `beforeId` = العنصر الذي سيقع **قبل** العنصر المُحرَّك (أعلاه/يمينه حسب الاتجاه).
- `afterId` = العنصر الذي سيقع **بعده**.
- أيٌّ منهما قد يكون `null` (يعني الطرف: البداية أو النهاية).

الخادم يُعيد قراءة مواضع هذين الجارين **لحظة التنفيذ داخل المعاملة** ويحسب المفتاح
الجديد بينهما بنفسه؛ لذا يُمنع أن يُرسل العميل سلسلة موضع محسوبة مسبقًا.

## سلوك الواجهة أثناء السحب (dnd-kit)

في `apps/web/src/features/boards/BoardPage.tsx`:
- القوائم داخل `SortableContext` أفقي؛ وبطاقات كل قائمة داخل `SortableContext` رأسي
  متداخل.
- كل عنصر قابل للسحب يحمل وسمًا `data: { type: 'list' | 'card' }` للتمييز بين "إعادة
  ترتيب الأعمدة" و"نقل بطاقة".
- القوائم الفارغة تحتاج منطقة إفلات مخصّصة (`${list.id}::empty`) لأن `SortableContext`
  الفارغ لا يملك عنصرًا يُسجَّل كهدف تحويم.
- الحالة المحلية تُحدَّث تفاؤليًا (Optimistic) أثناء السحب، ثم تُوفَّق مع الخادم عند
  الإفلات بإرسال معرّفات الجيران المحسوبة من المصفوفة المحلية — بما يطابق واجهة النقل في
  الخادم.

## التحقّق من الطلبات (Validation)

لا يُستخدم `ValidationPipe` الافتراضي في Nest؛ بل تُتحقَّق أجسام الطلبات بنفس مخطّطات
**zod** المُعرَّفة في `packages/types` عبر `ZodValidationPipe` مخصّص يُطبَّق لكل معامل:
`@Body(new ZodValidationPipe(SomeRequestSchema))`. الهدف: تعريف أشكال الطلب/الاستجابة
مرة واحدة يستوردها الطرفان.

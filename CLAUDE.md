This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Business logic documentation (`docs/`) — keep it in sync

The `docs/` folder contains **Arabic** documentation of the project's business
logic (domain model, auth, authorization, board/list/card rules, fractional-
index ordering, admin rules). It is the human-readable reference for *what the
system does and why*.

**These docs must stay in sync with the code.** Whenever you change business
logic or add a feature — a permission rule, a request/response shape, a schema
field, the ordering algorithm, a new endpoint, or any new logical decision —
update the relevant file in `docs/` **in the same change**. The change is not
complete until the docs are updated. Use the "where do I document what" map and
checklist in [`docs/08-maintaining-docs.md`](docs/08-maintaining-docs.md) to
find the right file; add a new `docs/NN-*.md` (and link it from
`docs/README.md`) when introducing a wholly new resource. Keep the docs in
Arabic, and keep code identifiers (file/function/field names) in English.

## Commands

This is a Turborepo + pnpm workspace monorepo. Run all commands from the repo root unless noted.

```bash
pnpm install                          # install all workspace deps

# Build shared packages BEFORE running the API — see "Workspace package
# build order" below. Root `pnpm dev`/`pnpm build` handles this via turbo's
# dependency graph, but ad-hoc commands (e.g. `node dist/main.js` directly)
# do not.
pnpm --filter @app/ordering build
pnpm --filter @app/types build

pnpm dev                              # runs apps/api (port 3000) + apps/web (port 5173) together
pnpm build                            # turbo build across all packages/apps
pnpm typecheck                        # turbo typecheck across all packages/apps
pnpm test                             # turbo test (currently only packages/ordering has real tests)

# Single-package/app commands
pnpm --filter @app/ordering test      # fractional-index unit tests (vitest)
pnpm --filter @app/api dev            # NestJS only (nest start --watch)
pnpm --filter @app/web dev            # Vite dev server only

# Run a single test file/case (from packages/ordering)
cd packages/ordering && npx vitest run src/index.test.ts
cd packages/ordering && npx vitest run -t "generates a key strictly between"

# Database (apps/api)
cd apps/api
cp .env.example .env                  # fill in real JWT secrets; DATABASE_URL for local Postgres
npx prisma migrate dev                # apply/create migrations (needs CREATEDB perms for the shadow DB)
npx prisma generate                   # regenerate Prisma client after schema.prisma changes
```

There is no Docker daemon assumption baked in — `docker-compose.yml` at the
root starts Postgres if Docker is available, but a native `postgresql` install
reachable at the `DATABASE_URL` in `apps/api/.env` works identically.

## Workspace package build order (important gotcha)

`packages/ordering` and `packages/types` are TypeScript-source packages that
compile to CommonJS `dist/` (see their `package.json` `main`/`types` fields
pointing at `dist/index.js`). **`apps/api` requires the compiled JS at
runtime** — it does not resolve `.ts` files from workspace deps directly.
If you edit either package and then run the API directly (`node dist/main.js`,
or `nest build` without first rebuilding the deps), you'll get
`Cannot find module '@app/ordering'` or similar. Always rebuild the changed
package (`pnpm --filter @app/ordering build`) before rebuilding/running
`apps/api`. Root `pnpm build`/`pnpm dev` via turbo handle this automatically
through the workspace dependency graph — prefer those over ad-hoc `nest
build` invocations in the `apps/api` directory.

Relatedly: if `tsc --noEmit` (typecheck) and `tsc` (build) are run against
the same package back-to-back, a stale `tsconfig.tsbuildinfo` from the
`--noEmit` run can cause the subsequent build to silently omit files it
thinks are unchanged. Delete `tsconfig.tsbuildinfo` in a package if a build
output is missing files that exist in `src/`.

## Architecture

Trello-style Kanban board (boards → lists → cards, drag-and-drop reorder).
Currently: NestJS API + React web app. Realtime (Socket.IO) and a mobile app
(Expo/React Native) are planned but not yet built — see README's Roadmap.

### Fractional-index positions (`packages/ordering`)

The single most important piece of correctness logic in the system. `List`
and `Card` rows store `position` as a lexicographically-sortable string
(LexoRank-style) rather than an integer index, generated by
`generateKeyBetween(a, b)` in `packages/ordering/src/index.ts`. Moving an item
only ever requires writing that one row — never renumbering siblings — which
is what makes drags cheap under concurrent edits and lets a future realtime
layer broadcast a single-row diff instead of a whole-list resync. This
package has no runtime dependencies and is unit-tested in isolation
(`src/index.test.ts`), including a simulated repeated-drag sequence that
asserts sort order and uniqueness hold after many moves. Both
`apps/api`'s move endpoints and any future client-side optimistic reordering
should go through this same algorithm — don't hand-roll positioning logic
elsewhere.

### Move/reorder request shape

Reorder requests reference **neighbor IDs**, not raw position strings:
`PATCH /lists/:id` and `PATCH /cards/:id` accept an optional
`move: { beforeId, afterId }` (see `packages/types/src/requests.ts`). The
service layer (`apps/api/src/lists/lists.service.ts`,
`apps/api/src/cards/cards.service.ts`) re-reads the current neighbor
positions fresh *inside a Prisma transaction* via
`computeMovePosition` (`apps/api/src/common/util/position.util.ts`) and
computes the new key server-side — a stale client-computed position can
never corrupt ordering. When moving a card across lists, `targetListId` can
differ from the card's current list; the service validates the target list
belongs to the same board and that any neighbor IDs actually belong to the
target list.

### Authorization: single source of truth

`BoardsService.assertMembership(userId, boardId, minRole)` in
`apps/api/src/boards/boards.service.ts` is the only place board access is
checked. `ListsModule` and `CardsModule` both import `BoardsModule` and call
through `BoardsService` rather than re-implementing membership checks —
`ListsService`/`CardsService` first resolve a list/card's `boardId`, then
delegate to this method. Any new board-scoped resource should follow the
same pattern instead of duplicating role logic. Roles are `OWNER | MEMBER`
(`BoardRole` in the Prisma schema and `packages/types`); archiving/deleting a
board and managing membership require `OWNER`, everything else requires
`MEMBER`.

### Auth

JWT access tokens (short-lived) + refresh tokens (long-lived, rotating).
`AuthService` (`apps/api/src/auth/auth.service.ts`) signs both on
login/register, storing only a SHA-256 hash of the refresh token
(`RefreshToken.tokenHash`) keyed by a `jti` UUID — never the raw token. Every
`/auth/refresh` call revokes the token it was called with and issues a new
one (rotation), so a leaked-then-reused refresh token is detectable/rejected.
`JwtStrategy` + `JwtAuthGuard` (`common/guards/jwt-auth.guard.ts`) protect
all non-auth routes; `@CurrentUser()` (`common/decorators/current-user.decorator.ts`)
pulls `{ id, email }` off the validated request.

### Request validation

Nest's built-in `ValidationPipe`/`class-validator` are **not** used. Request
bodies are validated with the same zod schemas from `packages/types` that
the frontend uses for its own DTOs, via a small custom
`ZodValidationPipe` (`apps/api/src/common/pipes/zod-validation.pipe.ts`)
applied per-parameter: `@Body(new ZodValidationPipe(SomeRequestSchema))`.
Keep request/response shapes defined once in `packages/types` and imported
by both sides rather than redefining DTOs in Nest-specific classes.

### Web app: dnd-kit multi-container drag-and-drop

`apps/web/src/features/boards/BoardPage.tsx` is the core of the frontend.
Lists are a `SortableContext` (horizontal strategy); each list's cards are a
nested `SortableContext` (vertical strategy). List/card sortable items carry
a `data: { type: 'list' | 'card' }` tag used throughout the drag handlers to
distinguish "reordering columns" from "moving a card" — check this before
adding new draggable types. Empty lists need a dedicated droppable
(`useDroppable` with id `${list.id}::empty`, `data: { type: 'list-dropzone' }`)
since an empty `SortableContext` has no card item to register a hover target
against; `resolveTargetListId()` in `BoardPage.tsx` is what reconciles all
three `over` shapes (`list`, `list-dropzone`, `card`) back to a target list
id. Local `lists` state is optimistically mutated in `onDragOver` (so the UI
reflows live during a drag) and reconciled with the server via mutations in
`onDragEnd`, which pass neighbor IDs computed from the just-reordered local
array — mirroring the server's neighbor-based move API.

The API client (`apps/web/src/lib/api-client.ts`) wraps `fetch`, attaches the
JWT access token, and on a 401 transparently calls `/auth/refresh` once
(de-duplicated via a shared in-flight promise so concurrent 401s don't each
trigger their own refresh) before retrying the original request.

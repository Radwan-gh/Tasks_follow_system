# 3. المصادقة (Authentication)

المصدر: `apps/api/src/auth/auth.service.ts` و `apps/api/src/auth/jwt.strategy.ts`
و `apps/api/src/common/guards/jwt-auth.guard.ts`، وفي الواجهة
`apps/web/src/lib/api-client.ts`.

## الفكرة العامة

يعتمد النظام على **رمزين**:

1. **رمز الدخول (Access Token / JWT):** قصير العمر (افتراضيًا 15 دقيقة). يُرسَل مع كل
   طلب في ترويسة `Authorization: Bearer ...`. يحمل داخله `sub` (معرّف المستخدم) و
   `email` و `role`.
2. **رمز التحديث (Refresh Token):** طويل العمر (افتراضيًا 30 يومًا). يُستخدم فقط للحصول
   على رمز دخول جديد عند انتهاء الأول، ويحمل داخله `sub` و `jti` (معرّف فريد للرمز).

سبب الفصل: رمز الدخول قصير العمر يقلّل خطر تسرّبه، ورمز التحديث يتيح بقاء المستخدم
مسجّلًا دون إعادة إدخال كلمة المرور.

## دورة الحياة (Endpoints)

| المسار | الوظيفة |
|---|---|
| `POST /auth/register` | إنشاء حساب جديد ثم إصدار الرمزين مباشرةً |
| `POST /auth/login` | التحقّق من البريد وكلمة المرور ثم إصدار الرمزين |
| `POST /auth/refresh` | إبطال رمز التحديث المُستخدَم وإصدار زوج جديد (تدوير) |
| `POST /auth/logout` | إبطال رمز التحديث الحالي |
| `GET /auth/me` | إرجاع بيانات المستخدم الحالي |

## قواعد العمل الأساسية

### التسجيل (Register)
- لا يُسمح بتكرار البريد الإلكتروني → يُرجَع خطأ `Conflict` إن كان مستخدمًا.
- كلمة المرور تُجزّأ بـ **bcrypt** بـ 12 جولة قبل الحفظ. لا تُخزَّن كلمة المرور الأصلية.

### تسجيل الدخول (Login)
- يُرفض الدخول عند عدم تطابق البريد أو كلمة المرور → `Unauthorized`.
- يُرفض الدخول إذا كان الحساب **غير مُفعّل** (`isActive = false`) → `Forbidden`.

### تدوير رموز التحديث (Rotation) — قاعدة أمان محورية
عند كل استدعاء لـ `/auth/refresh`:
1. يُتحقّق من توقيع الرمز، ثم يُبحث عنه في قاعدة البيانات عبر `jti`.
2. يُرفض إن كان: غير موجود، أو تجزئته لا تطابق المخزَّن، أو مُبطَلًا، أو منتهيًا.
3. يُعاد قراءة المستخدم من قاعدة البيانات؛ إن أصبح غير مفعّل يُرفض التحديث.
4. **يُبطَل الرمز المُستخدَم فورًا** ويُصدَر رمز جديد.

الأثر: لو تسرّب رمز تحديث واستُخدم، فإن أول استخدام يُبطله؛ وأي استخدام لاحق (سواء من
المهاجم أو المستخدم الأصلي) يُرفَض، ما يجعل إعادة الاستخدام **قابلة للكشف والرفض**.

### تخزين رموز التحديث بأمان
لا يُخزَّن الرمز الخام إطلاقًا؛ يُخزَّن فقط **تجزئة SHA-256** له (`RefreshToken.tokenHash`)
مرتبطة بمعرّف `jti`. لذا حتى لو تسرّبت قاعدة البيانات لا يمكن انتحال رموز التحديث.

### انتشار تغييرات الدور/التفعيل
بما أن رمز الدخول قد يحمل `role` قديمًا، فإن أي تغيير في دور المستخدم أو تفعيله يظهر
أثره **خلال مدة رمز الدخول على الأكثر** (لأن `/auth/refresh` يعيد قراءة المستخدم من
قاعدة البيانات). ولتقصير هذه المدة أكثر، عمليات الإدارة الحسّاسة تُبطِل رموز التحديث
مباشرةً (انظر [`07-admin.md`](./07-admin.md)).

## الحراسة (Guards) واستخراج المستخدم
- **`JwtAuthGuard`** يحمي كل المسارات ما عدا `/auth/*`.
- **`@CurrentUser()`** يستخرج `{ id, email, role }` من الطلب بعد التحقّق.

## سلوك الواجهة عند انتهاء رمز الدخول
عميل الـ API (`apps/web/src/lib/api-client.ts`) عند تلقّيه استجابة `401`:
- يستدعي `/auth/refresh` **مرة واحدة** ثم يعيد الطلب الأصلي.
- الاستدعاءات المتزامنة تشترك في **وعد (Promise) واحد قيد التنفيذ** حتى لا يطلق كل طلب
  تحديثًا منفصلًا.
